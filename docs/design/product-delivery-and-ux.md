## 一、产品定位与交付形态

### 1.1 产品定位

**一句话描述**: 面向科研人员的 AI 驱动文献综述助手 — 输入研究问题，自动完成检索、阅读、分析、写作全流程，交付可直接使用的学术综述。

**核心价值**: 将传统需要数天至数周的文献综述工作，压缩至数分钟至数小时的人机协作过程。

### 1.2 交付形态

产品以 **Web 应用** 为主要交付形态，辅以命令行接口供高级用户和自动化场景使用：

| 交付形态            | 目标用户          | 说明                                             |
| ------------------- | ----------------- | ------------------------------------------------ |
| **Web 应用 (主要)** | 所有用户          | 浏览器访问，无需安装，全功能图形化界面           |
| **CLI 工具 (辅助)** | 开发者 / 高级用户 | 命令行调用，适合批量任务、脚本集成、CI/CD 流水线 |
| **API 服务 (开放)** | 第三方集成        | RESTful API，允许其他工具/平台接入综述能力       |

### 1.3 访问方式

| 方式          | 说明                                                         |
| ------------- | ------------------------------------------------------------ |
| **私有部署**  | Docker Compose 一键部署到个人服务器/实验室内网，数据完全自控 |
| **云端 SaaS** | 未来可选，注册即用，按用量计费                               |

## 三、用户界面总体设计

### 3.1 设计原则

| 原则              | 说明                                                                                   |
| ----------------- | -------------------------------------------------------------------------------------- |
| **对话驱动**      | 以自然语言对话为主要交互方式，降低学习成本                                             |
| **渐进披露**      | 默认简洁，高级选项按需展开，不让用户被复杂参数淹没                                     |
| **过程可见**      | Agent 工作过程实时可见（正在检索→正在阅读→正在分析），增强信任感                       |
| **渐进式呈现**    | 长时间任务采用渐进式结果展示（每完成一篇论文精读即实时呈现），而非全部完成后一次性展示 |
| **Human-in-loop** | 关键节点暂停等待用户确认，而非全自动黑盒                                               |
| **结果可操作**    | 输出结果可直接编辑、导出、引用，无需二次加工                                           |
| **容错友好**      | 异常状态有清晰提示和恢复引导，不让用户陷入死胡同                                       |
| **成本透明**      | Token 消耗与费用实时可见，用户对成本有预期和控制权                                     |


**交互要素**:

- 最近项目显示状态标识（进行中/已完成），点击可恢复中断的项目
- 首次使用时展示简要引导（3 步 Onboarding：描述问题 → 确认论文 → 获取综述）

**交互要素**:
- **左侧对话流**: Agent 工作状态实时反馈，关键节点暂停等待用户决策；HITL 决策点以卡片样式突出显示（区别于普通消息）；Agent 状态更新可折叠（"Search Agent 完成 — 点击查看详情"）
- **右侧数据面板**: 根据当前阶段自适应内容（论文列表 → 渐进式精读结果 → 分析图表 → 综述预览）
- **论文信息增强**: 每篇论文展示发表会议/期刊、全文可获取状态（📄全文 / 📋仅摘要）、与研究问题的相关度评分、原文链接
- **成本预估卡片**: 确认继续前展示预估 Token 消耗和费用，用户可调整预算
- **渐进式精读**: Reader Agent 每完成一篇即实时展示该篇结构化摘要，无需等待全部完成
- **预估剩余时间**: 根据已完成论文数量和速率动态估算

**交互要素**:
- **左侧大纲导航**: 点击可跳转到对应章节，支持拖拽调整结构
- **引用验证标识**: 正文中每条引用显示验证状态（✅ 已验证 / ⚠️ 待确认），左侧面板汇总统计。待确认引用高亮警告，用户可一键查看详情或手动确认
- **右侧正文编辑**: 富文本编辑，支持直接修改 AI 生成的内容
- **内联 AI 指令**: 选中文字后可对特定段落下达修改指令

- **撤销/重做**: 支持 Ctrl+Z / Ctrl+Y，AI 修改后不满意可一键回退
- **对比视图**: 查看历史版本时可开启 diff 模式，高亮显示修改前后的差异
- **导出菜单**: Word / LaTeX / Markdown / PDF / BibTeX (.bib) / RIS (.ris)，含参考文献格式化

## 五、异常状态与容错体验

系统需为各类异常状态提供清晰的用户提示和恢复引导，不仅设计 Happy Path，还要覆盖 Unhappy Path：

```
│  当前结果可能不完整。您可以:                       │
```

## 六、典型用户使用流程

### 6.1 场景一：研究生开题综述

```
步骤 1: 新建项目
  └─ 输入: "深度学习在医学影像分割中的应用，侧重 Transformer 架构"
  └─ 选择输出类型: Full Review
  └─ 系统展示预估: "预计检索 ~50 篇，精读+综述约需 5-8 分钟，预估消耗 ~600K tokens"

步骤 2: 确认检索范围 (Human-in-loop #1)
  └─ 系统展示 58 篇候选论文（全文可获取 35 篇，仅摘要 23 篇）
  └─ 用户: 去掉 2020 年之前的，保留 42 篇

步骤 3: 渐进式精读
  └─ Reader Agent 逐篇精读，每完成一篇即实时展示结构化摘要
  └─ 状态栏: "Reader Agent 精读中 (18/42) — 预计剩余 ~2 分钟"
  └─ 用户可在精读过程中浏览已完成论文的结果

步骤 4: 查看分析看板
  └─ 浏览主题聚类、方法对比矩阵、研究趋势
  └─ 系统提示: "Critic Agent 发现'对比学习'方向论文不足，已自动补充 5 篇"
  └─ 用户确认补充结果

步骤 5: 确认综述大纲 (Human-in-loop #2)
  └─ 系统生成 6 章大纲
  └─ 用户: 合并第 3、4 章，增加一节 "临床应用案例"

步骤 6: 生成综述全文 + 引用验证
  └─ Writer Agent 按确认后的大纲生成完整综述
  └─ 系统自动验证 32 条引用: 30 条已验证，2 条标记为「待确认」
  └─ 用户在编辑器中微调表述，检查待确认引用

步骤 7: 导出
  └─ 选择 Word 格式 + GB/T 7714 引用格式 + BibTeX 参考文献
  └─ 下载文件，直接放入开题报告
```

### 6.2 场景二：科研人员快速调研

```
步骤 1: 输入: "multi-agent LLM 最新进展 2024-2025"
步骤 2: 选择输出类型: Quick Brief （首页输入区域直接选择）
步骤 3: 3 分钟后获得 2 页领域速览
  └─ 包含: Top 10 关键论文、3 个主要方向、核心结论
  └─ 底部消耗提示: "本次消耗 45K tokens ($0.07)"
步骤 4: 对某个方向感兴趣，追问: "展开第2个方向的详细综述"
  └─ 系统从已有结果派生子项目，无需重复检索
步骤 5: 获得该子方向的 Full Review
```

### 6.3 场景三：从已有文献库出发

```
步骤 1: 导入 Zotero 导出的 .bib 文件 (含 45 篇论文)
  └─ 或上传 3 篇 PDF 论文作为种子文献
步骤 2: 输入: "基于这些论文做一个系统性综述"
步骤 3: 系统解析导入的文献，通过引用链扩展到 60 篇相关论文
步骤 4: 查看分析看板中的引文网络图
步骤 5: 生成综述，导出为 LaTeX + IEEE 引用格式 + .bib 文件
```

### 6.4 场景四：中断恢复

```
步骤 1: 用户昨天创建了一个综述项目，做到精读阶段后关闭浏览器
步骤 2: 今天重新打开应用，首页显示 "🔄 发现未完成的项目"
步骤 3: 点击 [从断点继续]，直接恢复到精读进度 (28/42)
步骤 4: Reader Agent 继续精读剩余 14 篇，无需从头开始
```

## 九、MVP 阶段界面范围

MVP 阶段聚焦核心交互闭环，分阶段交付界面能力：

### 9.1 MVP v0.1 — 命令行界面

| 能力           | 说明                                              |
| -------------- | ------------------------------------------------- |
| 文本输入       | 命令行输入研究问题                                |
| 进度展示       | 终端打印 Agent 工作状态，逐篇展示精读结果         |
| 结果输出       | 生成 Markdown + BibTeX (.bib) 文件到本地          |
| 人工确认       | 终端交互式确认（Y/N / 选择列表）                  |
| 引用验证       | 输出中标注引用验证状态（[✅ 已验证] / [⚠️ 待确认]） |
| Token 消耗展示 | 终端打印已消耗 Token 数和预估费用                 |
| 全文覆盖率     | 展示 "全文精读 X 篇 / 仅摘要 Y 篇"                |
| BibTeX 导入    | 支持通过命令行参数传入 .bib 文件作为种子文献      |

**CLI 模式下 HITL 简化策略**:

CLI 不具备 Web 界面的富交互能力（如拖拽大纲、富文本编辑），因此 3 个 HITL 节点在 CLI 中采用简化交互模式：

| HITL 节点    | Web 界面                        | CLI 简化模式                                                                                           |
| ------------ | ------------------------------- | ------------------------------------------------------------------------------------------------------ |
| **检索确认** | 可勾选的论文列表，支持过滤/排序 | 打印编号列表 → 用户输入序号排除（如 `exclude 3,7,12`）或输入 `all` 全选，支持 `add "query"` 追加检索   |
| **大纲审阅** | 可拖拽调整的大纲树              | 打印大纲文本 → 用户输入 `yes` 通过 / `edit` 打开 `$EDITOR` 编辑大纲 YAML / `retry "指令"` 要求重新生成 |
| **初稿审阅** | 富文本编辑器 + 内联 AI 指令     | 打印综述到终端或自动打开预览 → 用户输入 `yes` 通过 / `revise "修改指令"` 要求修改                      |

CLI 示例交互：

```
🔍 Search Agent 找到 47 篇候选论文:

  #  | 年份 | 被引 | 全文 | 标题
  1  | 2024 |  128 |  📄  | Attention Is All You Need for Code...
  2  | 2023 |  356 |  📄  | Large Language Models for Software...
  3  | 2022 |   45 |  📋  | A Survey of Neural Code Generation...
  ...

>>> 操作 [all/exclude 序号/add "查询词"/done]: exclude 3,15
>>> 操作 [all/exclude 序号/add "查询词"/done]: done
✅ 已确认 45 篇论文，开始精读...
```

### 9.4 移动端适配策略

 Web 应用采用**响应式设计**，优先保障桌面端体验，移动端（手机/平板）提供「查看与审阅」能力：

| 场景         | 桌面端 (≥1024px) | 移动端 (<1024px)                      |
| ------------ | ---------------- | ------------------------------------- |
| 新建项目     | 完整表单         | 简化表单（仅研究问题 + 输出类型）     |
| 对话与工作流 | 双栏布局         | 单栏，数据面板收起为底部抽屉          |
| 论文列表     | 表格 + 过滤侧栏  | 卡片列表 + 顶部过滤器                 |
| 综述编辑     | 富文本编辑器     | 只读预览 + 简单评论，编辑引导至桌面端 |
| HITL 确认    | 完整交互         | 支持（简化为卡片式勾选 / 一键确认）   |
| 分析看板     | 图表可视化       | 图表堆叠展示，支持横屏                |

> **MVP 阶段**: 移动端适配为 P2 优先级，v0.2 Web 界面优先保障桌面端；v0.4 补齐移动端响应式布局。
