## 一、目标用户与核心场景

| 用户角色     | 典型场景                                                     |
| ------------ | ------------------------------------------------------------ |
| 研究生       | 开题前快速了解某领域全貌，撰写开题报告中的文献综述章节       |
| 科研人员     | 写论文 Related Work、追踪领域最新进展、发现研究空白          |
| 课题组负责人 | 评估新方向可行性、了解竞争格局                               |
| 跨学科研究者 | 快速进入陌生领域、找到领域间的交叉点                         |
| 本科生       | 毕业论文文献综述，需要更多引导，适合使用预设检索策略模板     |
| 企业研发人员 | 技术调研、专利与技术报告的文献分析，关注工程实践和技术可行性 |

## 二、多智能体架构设计

### 2.1 整体架构

```
用户输入 (研究问题 / 关键词 / 上传论文)
         │
         ▼
┌─────────────────────────────────┐
│      Orchestrator Agent         │  ← 意图解析、任务分解、调度、质量把控
│      (编排智能体 — LLM 驱动)    │
└─────┬───┬───┬───┬───┬───┬──────┘
      │   │   │   │   │   │
      ▼   ▼   ▼   ▼   ▼   ▼
┌───┐┌───┐┌───┐┌───┐┌───┐┌───┐
│ 1 ││ 2 ││ 3 ││ 4 ││ 5 ││ 6 │
└───┘└───┘└───┘└───┘└───┘└───┘
```

**Orchestrator 职责说明**: Orchestrator 使用 LLM 进行意图解析与动态策略生成（如根据用户问题自动选择最优检索策略），同时通过 LangGraph DAG 定义固定的阶段流转逻辑。它不是纯代码路由器，也不是完全自由的 LLM Agent，而是「LLM 决策 + 确定性编排」的混合模式。

### 2.2 六大专业智能体

| #   | 智能体                         | 职责                         | 核心能力                                                 |
| --- | ------------------------------ | ---------------------------- | -------------------------------------------------------- |
| 1   | **Search Agent (检索智能体)**  | 多源文献检索、查询扩展、去重 | 将用户自然语言转化为多数据库查询；迭代式检索（滚雪球法） |
| 2   | **Reader Agent (阅读智能体)**  | 全文/摘要精读、信息抽取      | 提取方法、数据集、结论、局限性；识别论文间引用关系       |
| 3   | **Analyst Agent (分析智能体)** | 聚类、趋势分析、对比         | 按主题/方法/时间聚类；构建知识图谱；识别研究演进脉络     |
| 4   | **Critic Agent (评审智能体)**  | 质量评估、矛盾检测、Gap 发现 | 评估证据强度；发现文献间矛盾结论；识别尚未解决的问题     |
| 5   | **Writer Agent (写作智能体)**  | 综述文本生成、多格式输出     | 学术写作风格；自动引用格式化；多粒度输出                 |
| 6   | **Update Agent (追踪智能体)**  | 新文献监控、增量更新         | 定期检索新发表论文；差异化报告；邮件/推送通知            |

## 三、功能模块详细设计

### 3.1 智能检索与发现

| 功能               | 描述                                                                                                                                            |
| ------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| **自然语言提问**   | 用户用自然语言描述研究问题，系统自动转为结构化查询                                                                                              |
| **多库并行检索**   | 同时查询多个学术数据库，合并去重                                                                                                                |
| **滚雪球检索**     | 从种子论文出发，沿引用链正向/反向扩展。设有终止条件：最大深度（默认 2 层）、单次扩展上限（默认 50 篇）、总候选上限（默认 200 篇），防止检索爆炸 |
| **查询建议与扩展** | 推荐同义词、相关概念、MeSH 术语等                                                                                                               |
| **筛选与过滤**     | 按年份、期刊等级、引用数、研究类型、开放获取等过滤                                                                                              |
| **相似论文推荐**   | 基于已选论文推荐语义相似的文献                                                                                                                  |

### 3.2 深度阅读与信息提取

| 功能               | 描述                                                                                                                                     |
| ------------------ | ---------------------------------------------------------------------------------------------------------------------------------------- |
| **全文获取与降级** | 优先通过 Unpaywall/arXiv/PMC 获取 OA 全文 PDF；无法获取全文时自动降级为摘要分析模式，并在结果中标注分析深度（全文/摘要），告知用户覆盖率 |
| **结构化摘要**     | 自动提取：目的、方法、数据集、主要发现、局限性                                                                                           |
| **关键概念标注**   | 识别并高亮论文中的关键概念、术语、定义                                                                                                   |
| **方法论提取**     | 提取实验设计、算法、模型架构等技术细节                                                                                                   |
| **数据集与基准**   | 汇总各论文使用的数据集、评估指标及性能数据                                                                                               |
| **论文间关系识别** | 标注论文之间的"扩展""反驳""复现""综述"等关系                                                                                             |
| **用户论文上传**   | 支持用户上传本地 PDF/BibTeX 文件，解析后纳入分析论文集，与在线检索结果统一管理                                                           |

### 3.3 分析与洞察

| 功能              | 描述                                                 |
| ----------------- | ---------------------------------------------------- |
| **主题聚类**      | 将文献自动分组为若干主题/子方向                      |
| **时间线分析**    | 展示领域发展的时间演进脉络                           |
| **方法对比矩阵**  | 生成各方法/模型在不同指标上的对比表格                |
| **引文网络分析**  | 构建引用关系图，识别关键节点论文（高被引、桥梁论文） |
| **作者/机构分析** | 识别核心研究团队、机构合作网络                       |
| **研究热度趋势**  | 基于发表量、引用量的趋势变化曲线                     |

### 3.4 研究空白与批判性分析

| 功能                  | 描述                                   |
| --------------------- | -------------------------------------- |
| **Research Gap 识别** | 基于现有文献自动推断尚未充分探索的方向 |
| **矛盾发现**          | 检测不同论文之间的矛盾结论并汇总       |
| **方法论局限汇总**    | 归纳现有研究的共性局限                 |
| **未来方向建议**      | 综合分析后给出可能的研究方向建议       |

### 3.5 写作与输出

| 功能               | 描述                                                                                                      |
| ------------------ | --------------------------------------------------------------------------------------------------------- |
| **自动生成综述**   | 按主题组织的完整文献综述文本                                                                              |
| **多风格写作**     | 支持叙述式 / 系统性 / 批判性等不同写作风格                                                                |
| **引用自动管理**   | 自动插入引用，支持 APA / IEEE / GB/T 7714 等格式                                                          |
| **大纲先行**       | 先生成大纲供用户调整，再填充内容                                                                          |
| **多语言输出**     | 中文 / 英文 / 双语对照                                                                                    |
| **引用验证**       | 每条引用回溯原始数据源确认论文存在性，防止 LLM 幻觉编造虚假引用                                           |
| **文献工具互操作** | 支持导入/导出 BibTeX (.bib) 和 RIS (.ris) 格式，与 Zotero / Mendeley / EndNote 等主流文献管理工具无缝衔接 |

### 3.6 持续追踪

| 功能           | 描述                                   |
| -------------- | -------------------------------------- |
| **新文献告警** | 定期检索并推送与关注主题相关的新论文   |
| **增量更新**   | 在已有综述基础上增补新发现的文献       |
| **版本管理**   | 综述的每次更新保留历史版本，可对比差异 |

## 四、输出结果类型

系统可以产出以下 **10 类交付物**，用户可按需选择：

| #   | 输出类型                                  | 格式                        | 说明                                                   |
| --- | ----------------------------------------- | --------------------------- | ------------------------------------------------------ |
| 1   | **快速摘要 (Quick Brief)**                | 1-2 页                      | 领域速览：核心问题、主要方法、关键论文 Top10、初步结论 |
| 2   | **注释文献列表 (Annotated Bibliography)** | 表格/列表                   | 每篇论文附 3-5 句点评，按主题分组                      |
| 3   | **完整文献综述 (Full Review)**            | 长文（Word/LaTeX/Markdown） | 引言→主题分析→对比讨论→研究空白→结论，含完整参考文献   |
| 4   | **对比矩阵 (Comparison Matrix)**          | 表格（Excel/CSV）           | 方法×指标 的系统对比表，适合论文插入                   |
| 5   | **知识图谱 (Knowledge Map)**              | 可交互图形                  | 概念、论文、作者之间的关系网络可视化                   |
| 6   | **时间线 (Timeline)**                     | 可视化图表                  | 领域发展里程碑式事件与关键论文的时序展示               |
| 7   | **研究空白报告 (Gap Report)**             | 结构化报告                  | 已解决问题 vs 待探索方向，附证据支撑                   |
| 8   | **趋势分析报告 (Trend Report)**           | 图文报告                    | 发文量趋势、热点关键词变迁、新兴主题预测               |
| 9   | **方法论综合报告 (Methodology Review)**   | 结构化报告                  | 各研究方法/技术路线的对比分析、适用场景、优劣势总结    |
| 10  | **研究脉络报告 (Research Roadmap)**       | 图文报告                    | 研究思路演进、方法迭代关系、关键转折点与启发性总结     |

## 五、文献数据库层设计

### 5.1 外部数据源接入

| 数据库               | 类型     | 覆盖领域           | API 可用性              | 风险与备注                          |
| -------------------- | -------- | ------------------ | ----------------------- | ----------------------------------- |
| **Semantic Scholar** | 开放     | 全学科             | ✅ 免费 API，含引用图谱  | 无 Key 限 1 req/s，建议申请 API Key |
| **OpenAlex**         | 开放     | 全学科 (2.5亿+)    | ✅ 完全免费，REST API    | 稳定，推荐作为主力元数据源          |
| **arXiv**            | 预印本   | CS/物理/数学/生物  | ✅ 免费 API              | 返回 Atom XML，需专用解析           |
| **PubMed / PMC**     | 生物医学 | 医学/生物/生命科学 | ✅ 免费 NCBI E-utilities | 稳定                                |
| **CrossRef**         | 元数据   | 全学科 DOI 元数据  | ✅ 免费 API              | 元数据补充用                        |
| **DBLP**             | CS       | 计算机科学         | ✅ 免费 API              | CS 领域补充                         |
| **CORE**             | 开放获取 | 全学科全文         | ✅ 免费 API              | 稳定性一般，作为备选                |
| **Unpaywall**        | OA 查找  | 查找论文 OA 版本   | ✅ 免费 API              | 全文获取辅助                        |

**数据源降级策略**: 当某个数据源不可用（超时/限流/宕机）时，系统自动跳过该源并使用其余可用数据源继续检索，在结果中标注实际使用的数据源列表。核心数据源（Semantic Scholar + arXiv）不可用时向用户发出警告，由用户决定是否继续。

## 六、用户交互流程

### 6.1 主流程

```
用户输入研究问题
    │
    ▼
[Orchestrator] 分析意图，制定检索策略
    │
    ├──▶ [Search Agent] 多库检索 → 候选论文集
    │         │
    │         ▼
    │    HITL: 用户确认论文列表 (可增删/调整检索)
    │         │
    │         ▼
    ├──▶ [Reader Agent] 批量精读 → 结构化信息
    │         │       │
    │         │       └──反馈──▶ [Search Agent] (发现高相关引用时触发补充检索)
    │         ▼
    ├──▶ [Analyst Agent] 聚类/对比/趋势 → 分析结果
    │         │
    │         ▼
    ├──▶ [Critic Agent] 质量评估/Gap 发现 → 批判报告
    │         │       │
    │         │       └──反馈──▶ [Search Agent] (文献覆盖不足时触发补充检索)
    │         ▼
    ├──▶ [Writer Agent] 生成大纲
    │         │
    │         ▼
    │    HITL: 用户审阅大纲
    │         │
    │         ▼
    ├──▶ [Writer Agent] 生成全文 + 引用验证
    │         │       │
    │         │       └──反馈──▶ [Reader Agent] (某观点缺乏支撑时触发定向精读)
    │         ▼
    │    HITL: 用户审阅初稿
    │         │
    │         ▼
    └──▶ [Writer Agent] 修订 + 格式化导出
              │
              ▼
         最终交付物
```

**反馈环路说明**: 工作流并非严格线性流水线，而是支持条件反馈边。当下游 Agent 发现信息不足时，可触发上游 Agent 定向补充，形成闭环迭代。每次反馈触发设有最大迭代次数限制（默认 2 次），防止无限循环。

### 6.2 关键交互节点（Human-in-the-loop）

1. **检索后确认** — 展示候选论文列表（含全文可获取状态标注），用户可增删、调整检索方向
2. **大纲审阅** — 生成综述大纲后暂停，等待用户确认/调整结构
3. **初稿审阅** — 生成初稿后，用户可要求扩展某部分、换写作风格或触发补充检索

### 6.3 成本透明

系统在以下环节向用户展示预估/实际 Token 消耗与成本：
- **启动前**: 根据检索结果数量和输出类型，预估 Token 消耗和费用
- **执行中**: 实时展示已消耗 Token 数
- **完成后**: 汇总本次综述的总消耗

用户可设置单次综述的 Token 预算上限，超限时暂停并询问是否继续。

## 七½、非功能性需求

> **说明**: 本节定义系统在性能、容量和可靠性方面的约束目标，为架构设计和性能测试提供基准。

### 性能目标

| 指标                  | 目标值             | 说明                                                                        |
| --------------------- | ------------------ | --------------------------------------------------------------------------- |
| **端到端综述生成**    | 30 篇论文 ≤ 8 分钟 | 含检索 + 精读 + 写作全流程（不含 HITL 等待时间），Quick Brief 目标 ≤ 3 分钟 |
| **50 篇论文综述**     | ≤ 15 分钟          | 大规模综述的上界目标，受 LLM API 速率限制约束                               |
| **检索响应**          | ≤ 10 秒            | Search Agent 多源并行检索到返回候选列表的时间                               |
| **单篇论文精读**      | ≤ 30 秒            | Reader Agent 对单篇论文的 LLM 分析耗时                                      |
| **API 响应 (非流式)** | P95 ≤ 500ms        | 项目 CRUD、状态查询等常规 REST 接口                                         |
| **SSE 首事件延迟**    | ≤ 2 秒             | 工作流启动后到第一个 SSE 事件推送到客户端                                   |

### 容量约束

| 约束项                 | MVP 目标      | 远期目标       | 说明                                         |
| ---------------------- | ------------- | -------------- | -------------------------------------------- |
| **最大并发项目数**     | 3 个          | 按 Worker 扩展 | 受 Celery Worker 数量和 LLM API 速率限制约束 |
| **单项目最大论文数**   | 200 篇        | 500 篇         | 由滚雪球检索候选上限决定                     |
| **单篇论文全文大小**   | ≤ 5MB (PDF)   | ≤ 10MB         | 超大 PDF 跳过全文解析，降级为仅摘要          |
| **用户上传文件大小**   | 单文件 ≤ 20MB | 单文件 ≤ 50MB  | BibTeX/RIS 文件通常较小，主要约束 PDF        |
| **单次综述最大 Token** | 200 万 tokens | 可配置         | 超限时暂停并询问用户（见 §6.3 成本透明）     |
| **数据库存储**         | ≤ 10GB        | 按需扩展       | SQLite 单文件，含论文元数据和解析文本        |

### 可靠性要求

| 要求             | 说明                                                                             |
| ---------------- | -------------------------------------------------------------------------------- |
| **断点恢复**     | 系统崩溃后可从最近的 LangGraph checkpoint 恢复，丢失不超过 1 个 Agent 步骤的工作 |
| **数据源容错**   | 任一非核心数据源不可用时，系统仍可正常运行并在结果中标注实际使用的数据源         |
| **LLM 调用重试** | 外部 API 调用失败自动重试（指数退避，最多 3 次），连续失败后降级或报错           |
| **数据持久性**   | 工作流每步完成后自动 checkpoint；最终结果回写业务数据库；用户上传的文件持久存储  |

## 八、风险评估与应对

| 风险类型         | 风险描述                                              | 影响                                                  | 应对策略                                                                                                                                                               |
| ---------------- | ----------------------------------------------------- | ----------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **LLM 幻觉**     | LLM 可能编造不存在的论文引用（虚假标题、作者、DOI）   | 严重破坏学术可信度，是文献综述场景的致命问题          | 实施引用验证机制：Writer Agent 输出的每条引用，回溯原始数据源（Semantic Scholar / CrossRef / DOI）确认论文存在性。验证失败的引用标记为「待确认」并在输出中高亮警告用户 |
| **全文不可获取** | 大多数学术论文非开放获取，仅靠 OA 渠道覆盖率有限      | Reader Agent 精读退化为摘要分析，分析深度低于用户预期 | 自动降级为摘要分析模式；在结果中标注每篇论文的分析深度（全文/摘要）；汇总全文覆盖率告知用户                                                                            |
| **检索爆炸**     | 滚雪球检索沿引用链指数级增长，Token 消耗失控          | 系统性能下降，成本不可控                              | 设置滚雪球深度上限（2 层）、单次扩展上限（50 篇）、总候选上限（200 篇）                                                                                                |
| **数据源不可用** | 外部 API 限流、超时或宕机                             | 检索结果不完整                                        | 数据源降级策略：自动跳过不可用源，核心源不可用时警告用户                                                                                                               |
| **Token 成本**   | 30+ 篇论文精读 + 长文生成，单次综述消耗可达百万 Token | 用户产生超预期费用                                    | Token 预算管理：启动前预估、执行中实时展示、超限暂停询问                                                                                                               |

## 九、MVP（最小可行产品）建议

第一阶段聚焦 **核心闭环**，建议实现：

| 优先级 | 内容       | 范围                                               |
| ------ | ---------- | -------------------------------------------------- |
| P0     | 核心 Agent | Search Agent + Reader Agent + Writer Agent         |
| P0     | 数据源     | Semantic Scholar + arXiv                           |
| P0     | 输出格式   | Quick Brief + Annotated Bibliography + Full Review |
| P0     | 基础存储   | SQLite + Chroma                                    |
| P0     | 引用验证   | 每条引用回溯数据源确认存在性，防止 LLM 幻觉        |
| P0     | 文献互操作 | 支持 BibTeX (.bib) / RIS (.ris) 导入导出           |
| P1     | 用户界面   | 命令行 / 简易 Web 界面                             |
| P1     | 成本透明   | Token 消耗预估与实时展示                           |

**MVP 输出类型说明**: Methodology Review 和 Research Roadmap 依赖 Analyst/Critic Agent 的分析能力（方法对比、研究演进脉络），MVP 阶段仅有 Search + Reader + Writer 三个 Agent，无法保障这两类输出的质量，因此延至 Analyst/Critic Agent 上线后作为 P1 开放。
